# Analytics API

Endpoints for retrieving salon revenue analytics and financial metrics. Salon owners can view analytics for their own salons, while admins can view analytics for any salon.

---

## Get Salon Analytics

**Description:** Retrieve comprehensive revenue analytics for a specific salon including total revenue, transactions, profit calculations, and category-wise spending breakdown. The system calculates metrics from both product orders (via payments) and service bookings.

**Endpoint:** `GET /api/v1/salons/:salonId/analytics`

**Authentication:** Required (Bearer token)

**URL Parameters:**

- `salonId` (string, required) — Salon ID in CUID format

**Query Parameters:**

- `startDate` (string, optional) — Start date for analytics in ISO 8601 datetime format (e.g., `2026-01-01T00:00:00Z`)
- `endDate` (string, optional) — End date for analytics in ISO 8601 datetime format (e.g., `2026-01-31T23:59:59Z`)

**Success Response (200 OK):**

```json
{
  "message": "Analytics retrieved successfully",
  "data": {
    "summary": {
      "totalRevenue": "12500.00",
      "totalTransactions": 45,
      "netProfit": "12250.00",
      "adminCommission": "250.00",
      "averageTransactionValue": "390.63",
      "uniqueCustomers": 32
    },
    "productRevenue": {
      "total": "7500.00",
      "byCategory": [
        {
          "category": "Products",
          "revenue": "7500.00",
          "count": 25,
          "percentage": 100.00
        }
      ]
    },
    "serviceRevenue": {
      "total": "5000.00",
      "byCategory": [
        {
          "category": "Haircut",
          "revenue": "3000.00",
          "count": 15,
          "percentage": 60.00
        },
        {
          "category": "Coloring",
          "revenue": "2000.00",
          "count": 5,
          "percentage": 40.00
        }
      ]
    },
    "timeRange": {
      "startDate": "2026-01-01T00:00:00.000Z",
      "endDate": "2026-01-31T23:59:59.999Z"
    }
  }
}
```

**Response Fields Explained:**

**Summary Metrics:**
- `totalRevenue` (string) — Total revenue from both products and services (Decimal formatted as string with 2 decimal places)
- `totalTransactions` (number) — Count of all completed transactions (succeeded payments + completed/confirmed bookings)
- `netProfit` (string) — Revenue after deducting 2% admin commission (totalRevenue - adminCommission)
- `adminCommission` (string) — 2% platform commission calculated from total revenue
- `averageTransactionValue` (string) — Average revenue per unique customer (totalRevenue / uniqueCustomers)
- `uniqueCustomers` (number) — Count of unique users who made purchases or bookings (deduplicated across both revenue streams)

**Product Revenue:**
- `total` (string) — Total revenue from product orders
- `byCategory` (array) — Breakdown by product category:
  - `category` (string|null) — Category name (currently "Products" for all items until Product model includes category field)
  - `revenue` (string) — Revenue generated by this category
  - `count` (number) — Number of transactions in this category
  - `percentage` (number) — Percentage of total product revenue (2 decimal places)

**Service Revenue:**
- `total` (string) — Total revenue from service bookings
- `byCategory` (array) — Breakdown by service category:
  - `category` (string|null) — Service category name (e.g., "Haircut", "Coloring", or "Uncategorized" if null)
  - `revenue` (string) — Revenue generated by this service category
  - `count` (number) — Number of bookings in this category
  - `percentage` (number) — Percentage of total service revenue (2 decimal places)

**Time Range:**
- `startDate` (string|null) — Start date of the analytics period (ISO 8601 format) or null if no filter applied
- `endDate` (string|null) — End date of the analytics period (ISO 8601 format) or null if no filter applied

## Validation Rules

### URL Parameters

- **salonId**: Must be a valid CUID format (e.g., `cmiplyw1n0002li2gbggmr85q`)

### Query Parameters

- **startDate** (optional):
  - Must be a valid ISO 8601 datetime string (e.g., `2026-01-01T00:00:00Z`)
  - Must be before or equal to `endDate` if both are provided
  - If omitted, no start date filter is applied (returns all historical data)

- **endDate** (optional):
  - Must be a valid ISO 8601 datetime string (e.g., `2026-01-31T23:59:59Z`)
  - Must be after or equal to `startDate` if both are provided
  - If omitted, no end date filter is applied (includes all data up to present)

**Date Range Validation:**
- Both `startDate` and `endDate` are optional
- If both are provided, `startDate` must be ≤ `endDate`
- Invalid date formats will return a 400 Bad Request error
- Date filtering is applied to `Payment.createdAt` for product orders and `Booking.createdAt` for service bookings

## Authentication & Authorization

### Authentication Requirements

- **Required**: All analytics endpoints require JWT authentication
- **Header Format**: `Authorization: Bearer <JWT_TOKEN>`
- Token must be obtained from the `/api/v1/auth` endpoints (login or register)

### Authorization Rules

**Salon Owners:**
- Can view analytics **only** for salons they own
- Ownership is verified by checking `Salon.ownerId` against the authenticated user's ID
- Attempting to access other salons' analytics returns `403 Forbidden`

**Admins:**
- Can view analytics for **any** salon in the system
- Admin role is determined by `user.role === 'admin'` or `user.role === 'ADMIN'`
- Must still provide a valid salonId that exists in the system

**Authorization Flow:**
1. User authentication is verified via JWT token
2. If user is not an admin:
   - System checks if user owns the requested salon
   - Returns 403 if ownership verification fails
3. If user is an admin:
   - System verifies salon exists
   - Returns 404 if salon doesn't exist
4. Analytics data is retrieved and returned

## Revenue Calculation Details

### Product Revenue

**Data Source:** `Payment` model with `status = 'succeeded'`

**Calculation:**
- Only payments with `succeeded` status are counted
- Excludes `pending`, `processing`, `failed`, `canceled`, and `refunded` payments
- Revenue is summed from `Payment.amount` field
- Filtered by `Order.salonId` to ensure attribution to correct salon
- Date filtering applied to `Payment.createdAt`

**Category Breakdown:**
- Currently, all products are grouped under "Products" category
- **Note:** Product model does not yet have a `category` field
- Future enhancement: Add category field to Product model for detailed categorization

### Service Revenue

**Data Source:** `Booking` model with `status IN ('COMPLETED', 'CONFIRMED')`

**Calculation:**
- Only bookings with `COMPLETED` or `CONFIRMED` status are counted
- Excludes `CANCELLED` and `NO_SHOW` bookings
- Revenue calculated from `Service.price` associated with each booking
- Filtered by `Booking.salonId`
- Date filtering applied to `Booking.createdAt`

**Important Note:** Bookings do not have explicit payment tracking in the current schema. The system assumes all `COMPLETED` and `CONFIRMED` bookings are paid and calculates revenue from the associated service price.

**Category Breakdown:**
- Services are grouped by `Service.category` field
- Services with `null` category are shown as "Uncategorized"
- Each category includes revenue amount, transaction count, and percentage of total service revenue

### Admin Commission

**Calculation:**
- Fixed rate: **2% of total revenue**
- Formula: `adminCommission = totalRevenue × 0.02`
- Applied to combined revenue (products + services)
- Deducted from total revenue to calculate net profit

**Example:**
```
Total Revenue: $10,000.00
Admin Commission (2%): $200.00
Net Profit: $9,800.00
```

### Average Transaction Value

**Calculation:**
- Formula: `averageTransactionValue = totalRevenue / uniqueCustomers`
- Counts unique customers across both revenue streams (deduplicated)
- Returns `0.00` if no customers (prevents division by zero)

**Example:**
```
Total Revenue: $12,500.00
Unique Customers: 32
Average Transaction Value: $390.63
```

### Unique Customer Count

**Deduplication Logic:**
- Combines customers from product orders (`Order.userId`) and service bookings (`Booking.userId`)
- Deduplicates using Set data structure
- A customer who both purchased products AND booked services is counted once

## Edge Cases & Behavior

### Deleted Services

**Scenario:** A booking references a service that has been deleted from the database.

**Behavior:**
- Booking is skipped from revenue calculations
- Warning is logged to console: `"Booking {id} has deleted service - skipping from revenue"`
- Does not cause API errors
- Transaction count is not incremented for these bookings

### Refunded Payments

**Scenario:** A payment that was previously succeeded has been refunded.

**Behavior:**
- Payments with `status = 'refunded'` are excluded from revenue calculations
- Only `succeeded` status payments are counted
- Ensures revenue reflects actual collected amounts

### Multiple Payments per Order

**Scenario:** An order has multiple payment attempts (e.g., failed payment followed by successful payment).

**Behavior:**
- Only payments with `status = 'succeeded'` are included
- Failed, pending, and canceled payments are ignored
- Prevents double-counting and ensures accurate revenue

### Empty Date Range

**Scenario:** No transactions exist within the specified date range.

**Behavior:**
- Returns zeros for all numeric metrics
- Returns empty arrays for category breakdowns
- Does not return an error
- Example response:
```json
{
  "summary": {
    "totalRevenue": "0.00",
    "totalTransactions": 0,
    "netProfit": "0.00",
    "adminCommission": "0.00",
    "averageTransactionValue": "0.00",
    "uniqueCustomers": 0
  },
  "productRevenue": {
    "total": "0.00",
    "byCategory": []
  },
  "serviceRevenue": {
    "total": "0.00",
    "byCategory": []
  },
  "timeRange": {
    "startDate": "2026-01-01T00:00:00.000Z",
    "endDate": "2026-01-31T23:59:59.000Z"
  }
}
```

### No Date Filters

**Scenario:** Request made without `startDate` or `endDate` parameters.

**Behavior:**
- Returns analytics for **all time** (entire history)
- No date filtering applied to database queries
- `timeRange` in response shows `null` for both dates:
```json
{
  "timeRange": {
    "startDate": null,
    "endDate": null
  }
}
```

### Decimal Precision

**Behavior:**
- All monetary values use `Prisma.Decimal` type for precise financial calculations
- Avoids JavaScript floating-point precision errors
- All decimal values returned as strings with 2 decimal places (e.g., `"12500.00"`)
- Ensures accurate commission calculations and revenue totals

## Performance Considerations

### Query Optimization

**Parallel Execution:**
- Product payments and service bookings are queried in parallel using `Promise.all()`
- Reduces total query time compared to sequential execution

**Selective Field Retrieval:**
- Only required fields are selected in database queries
- Minimizes data transfer between database and application
- Example: Only `id`, `userId`, `price`, `category` are fetched for services

**Database Indexes:**
- Existing indexes are leveraged for efficient queries:
  - `Payment.orderId` (indexed)
  - `Order.salonId` (indexed)
  - `Booking.salonId` (indexed)
  - `Service.category` (indexed)

### Response Time Expectations

**Typical Response Times:**
- Small datasets (< 100 transactions): 50-200ms
- Medium datasets (100-1000 transactions): 200-500ms
- Large datasets (> 1000 transactions): 500-1500ms

**Optimization Recommendations:**
- Use date range filters to limit dataset size
- Consider implementing caching for frequently accessed analytics (future enhancement)
- For very large salons, consider aggregating analytics in background jobs

## Technical Implementation Notes

### Architecture

**File Structure:**
```
src/
├── types/analytics.ts           → TypeScript interfaces
├── schemas/analyticsSchema.ts   → Zod validation schemas
├── services/analyticsService.ts → Business logic & calculations
├── controllers/analyticsController.ts → HTTP handlers
└── routes/analyticsRoute.ts     → Route definitions
```

**Technology Stack:**
- **ORM:** Prisma (type-safe database queries)
- **Validation:** Zod (runtime schema validation)
- **Decimal Math:** Prisma.Decimal (precise financial calculations)
- **Authentication:** JWT (JSON Web Tokens)
- **Date Handling:** Native JavaScript Date objects

### Data Flow

```
HTTP Request
    ↓
[analyticsRoute.ts]
    ↓
[Authentication Middleware] → Verify JWT token
    ↓
[Validation Middleware] → Validate params & query with Zod
    ↓
[analyticsController.ts]
    ↓
[Authorization Check] → Verify salon ownership or admin role
    ↓
[analyticsService.ts]
    ↓
[Database Queries] → Fetch payments & bookings (parallel)
    ↓
[Revenue Calculation] → Sum amounts, group by category
    ↓
[Commission Calculation] → Apply 2% commission
    ↓
[Response Formatting] → Convert Decimals to strings, structure JSON
    ↓
HTTP Response
```

### Error Handling

**Validation Errors (400):**
- Zod schema validation failures
- Invalid CUID format
- Invalid ISO 8601 dates
- startDate > endDate

**Authentication Errors (401):**
- Missing JWT token
- Invalid/expired JWT token
- Token verification failure

**Authorization Errors (403):**
- Non-admin user attempting to access another salon's analytics
- Ownership verification failure

**Not Found Errors (404):**
- Salon ID does not exist in database
- Only for admin requests (salon owners get 403 if not owner)

**Server Errors (500):**
- Database connection failures
- Unexpected errors during calculation
- Logged to console for debugging

## Security Considerations

### Data Privacy

**Salon Isolation:**
- Each salon's financial data is strictly isolated
- Non-admin users cannot access other salons' data
- Authorization verified on every request (no session-based access)

**PII Protection:**
- Customer personally identifiable information (PII) is not exposed
- Only aggregated metrics and counts are returned
- Individual transaction details are not included in response

### Rate Limiting

**Recommendations:**
- Apply rate limiting to prevent abuse (e.g., 100 requests per hour per user)
- Consider stricter limits for unauthenticated endpoints (if any are added)
- Monitor for unusual access patterns (admin accessing many salons rapidly)

### Audit Logging

**Recommended Logging:**
- Log all analytics access with:
  - User ID
  - Salon ID
  - Timestamp
  - Date range requested
  - IP address
- Enables compliance with financial data access regulations
- Helps detect unauthorized access attempts

### SQL Injection Protection

**Prisma ORM:**
- All queries use Prisma's parameterized query system
- Automatically prevents SQL injection attacks
- User input is never directly interpolated into SQL strings

## Troubleshooting

### Common Issues

**Issue:** "Unauthorized: You do not own this salon"
- **Cause:** User is trying to access another salon's analytics
- **Solution:** Verify you're using the correct salonId for your salon
- **For Admins:** Ensure your user role is set to 'admin' or 'ADMIN'

**Issue:** "Salon not found"
- **Cause:** salonId does not exist in database
- **Solution:** Verify the salonId is correct and exists
- **Check:** Run `GET /api/v1/salons` to list available salons

**Issue:** "startDate must be before or equal to endDate"
- **Cause:** Date range is reversed
- **Solution:** Swap the dates or correct the typo

**Issue:** Empty revenue despite having transactions
- **Cause:** Transactions outside the specified date range
- **Solution:** Expand date range or remove filters to see all-time data
- **Check:** Verify `Payment.createdAt` and `Booking.createdAt` dates

**Issue:** Service revenue showing as 0.00 despite completed bookings
- **Cause:** Bookings may have status other than 'COMPLETED' or 'CONFIRMED'
- **Solution:** Check booking statuses in database
- **Note:** Only 'COMPLETED' and 'CONFIRMED' bookings count as revenue

**Issue:** Category breakdown percentages don't sum to 100%
- **Cause:** Rounding precision (normal behavior)
- **Solution:** This is expected due to rounding to 2 decimal places
- **Note:** Each revenue stream (products/services) totals to 100% independently
